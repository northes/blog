RAG（Retrieval-Augmented Generation），中文名 检索增强生成

LLM 的应用场景中如 企业知识库、客服机器人、文档助手 的背后基本都有 RAG 技术的应用

## 为什么需要 RAG

现如今的 LLM 已经非常强大了，为什么还需要 RAG 技术

LLM 的本质是在预测下一个 TOKEN，这会带来三个问题

1. LLM 幻觉：训练数据中没有答案时，不会直接说不知道，而是会继续向后预测下一个 Token，直到输出一个看起来合理的答案
2. 知识过时：LLM 训练时的数据集的时间停留在 2025 年 12 月，那么对于 2026 年和之后的发生的事情，LLM 就会不知道
3. 无法访问私有数据：企业的内部数据，是不会公开到公网上的，此时模型的训练数据中根本不会有企业内部数据的知识，此时就需要我们主动提供给
   LLM 进行阅读和生成

这时候我们就会自然而然的想到，直接在输入框中将整个知识库文档丢给 LLM，让他从里面找我们想要的知识不就完了吗？

但是这里面又会存在问题：

1. LLM 的上下文是有限的，当超出 LLM 的上下文限制，输入的内容可能会被丢弃或压缩，这时候就有可能导致丢失了关键信息，导致给出了错误的答案
2. LLM 过大的上下文会导致每次对话的成本激增，我们知道每次向 LLM
   发起请求时，都会把之前对话过的内容拼接上再请求，这就会导致大量无用的信息被掺杂进去，浪费了大量的成本，无论是金钱还是性能
3. 回复速度过慢，如果把整个知识库都丢进去，每次向 LLM 提问时，LLM 都需要把整个知识库完整读一遍才能回答你的问题，这就会导致用户需要在那里盯着白屏等好几分钟，造成了极差的用户体验

所以最好的方式就是先在知识库中找到最相关的那几页或者哪几段内容，再丢给
LLM，就像开卷考试，我们不可能每次都从头到尾翻整本书找答案，正常都是直接定位到大概的章节和哪几页，再在那几页中找到正确答案

因此，我们可以知道，我们需要先检索，然后把检索出来的内容交给 LLM 进行生成，这就是需要 RAG 技术的原因，这也是 检索增强生成
的最直观的解答

## RAG 和核心流程

### 第一阶段：数据准备

假设我们需要给企业做一个问答知识库

1. 企业内部的文档通常存在各种各样的格式：如 PDF、Word、Excel、PPT、Markdown
   等，这些文档里可能混杂着各种各样的可能无关的数据，看起来就很“脏”，而“脏”的东西我们一般会对其进行清洗后再使用，对应到这些文档上，我们首先要做的就是对这些文档进行“数据清洗”，去除掉其中的无用的信息。

2. 对这些文档进行切分(chunking)，将文档按照章节、段落、字数等切分成小块(chunk)
   ，每个小块表达一个相对完整的语义，如“张三是一个好男人”，如果我们按照“张三”“是一个好男人”
   来切分，就会完全都是掉原句的语义，我们不知道张三的性别、物种，也不知道“是一个好男人”指的是谁。

> 具体使用哪种切分方式，需要根据实际的情况进行选择

> 为什么需要切分？如果按照整个文档进行处理，会导致后续召回的也是一个完整的文档，无法定位到具体 的信息，失去了 RAG 的作用

3. 由于计算机是不认识文字的，他只认识数字，因此我们需要对这些块进行向量化(embedding)，向量化其实就是把一句话通过 embedding
   模型，转成一组语义空间中的坐标点，举个例子


- “张三是男的”，转成坐标点就是（1,1）
- “张三三十岁”，转成坐标点就是（2,1）
- “今天天气很好”，转成坐标点就是（-2,-1）

在这个空间中，语义相近语句坐标会更接近，比如“张三是男的”和“张三三十岁”，都是描述张三这个人的一些信息，坐标就会更接近

在实际的 embedding 模型中，实际的坐标不会是这种二维的坐标，而是上百甚至几千的维度，在我们现实生活中画不出来，但是在数学上是存在的

将每一个 chunk 都通过 embedding 模型转成向量后，我们把文本的内容和对应的向量存入向量数据库中，举个简单的例子

|  向量坐标   |   文本   |
|:-------:|:------:|
|  (1,1)  | 张三是男的  |
|  (2,1)  | 张三三十岁  |
| (-2,-1) | 今天天气很好 |
|   ...   |  ...   |

### 第二阶段：用户提问

用户提出一个问题：张三是谁

在非 RAG 的场景下，LLM 可能会根据训练的数据进行编造一个虚假的张三出来

在 RAG 场景下，LLM 识别到需要调用 RAG 的工具时，会调用 RAG 工具

> 关于 LLM 如何调用工具，请看下集

同样的，在 RAG 工具的内部，会对传入的用户的问题进行向量化（embedding），举个例子，用户提问：张三是谁？就会将这个问题向量化成（1.5,0.5），参考上图的橙色坐标点

向量化后的文本会跟向量数据库中的数据的向量坐标进行计算，通常会采用的相似度几双方法有：余弦值、欧式距离

欧式距离：直接计算问题坐标和其余坐标之间的直线距离

计算后，返回其中最相关的 TopK 个块，K 是数学中的常数，可以等于 1，也可以等于 100，假设这里的 K 是 10，那么就召回 10 个最相关的片段，注意，到这里只是粗筛，只能判断召回的片段和我们的问题像不像，但不能保证这 10 个片段到底有没有回答这个问题

因此，这里就需要进行重排序（rerank），即使用更精确的方法，去逐一对比问题和召回的片段，主要是判断召回的这 10 个片段是否真的回答了这个问题

经过重排序后，我们可以得到 3 个跟用户的提问最相关的片段，那么我们就将这 3 个片段、用户的问题、基础提示词（prompt）等组合成一个新的提示词交给 LLM 进行回答

如

```
Answer the following question using only the provided document excerpts.
Question: <用户的问题>
Document excerpts: 
1. <重排后的片段 1>
2. <重排后的片段 2>
3. <重排后的片段 3>
Provide a clear, concise answer. Cite excerpt numbers like [1] when referencing sources.
```

需要注意的是，上面提到的将用户的问题进行向量化、查询向量数据库召回、重排都是发生在被调用的 RAG 工具中，对用户而言是透明的


用户眼中只有：发送问题 -> 获得答案
实际的调用：取得用户的问题 -> 对问题进行向量化 -> 查询向量数据库以召回 -> 重排 -> 将新的prompt 交给 LLM 进行回答 -> 流式输出给用户

至此可以发现，RAG 没有改变 LLM 本身，只是给 LLM 加入了一个可以查询外部数据的工具

整个系统分工明确：
- embedding 向量化
- 向量搜索 很快，粗筛
- rerank 精搜，打分，较慢
- LLM 生成答案

## 坑和优化技巧

1. 以企业知识库为例，企业的私有文档有很多格式，并且其中会有很多无用的噪声文本，在对知识库进行embedding 之前，需要花费大量的精力去进行数据清洗，包括但不限于 OCR、正则匹配、重写等技术或非技术手段进行处理
2. 块的切分大小，是最关键的点之一，如果我们的 chunk 分得非常大，就会导致一个 chunk 里包含的内容非常多，检索不准确，并且可能分散 LLM 的注意力，如果 chunk 分得非常小，就会导致语义断裂，比如“张三是个好人”，切分成“张三”“是个好人”，就会丢失张三的描述、且不知道描述好人的主体是谁。我们需要保证每一个块的语义都是相对完整的，，即信息的颗粒度要刚好，在具体场景中具体分析
3. 用户提问，会比较口语化，需要对输入的问题进行 Query Rewrite 问题重写，比如用户提问“张三是谁”，我们需要重写为“列出张三的性别、年龄、身高、视力、评价、人物关系信息”，核心目的是为了更方便后续的向量搜索
4. 在 RAG 检索的过程中，同样可以进行优化，常用的是混合检索（关键词检索 + 向量相似度检索），关键词检索就很像我们查字典，根据关键字去精确匹配。互补两者的盲区。
5. 接入 trace，并提供反馈功能，如果用户对某个回答不满意，可以通过反馈按钮提交当前对话的 id 信息，我们可以根据 trace 去查具体是 embedding 的问题、召回的问题、rerank 的问题还是 LLM 的生成问题，从而优化整个流程。补充：可以拓展下 harness engineer
6. 模型基座的选择：
   1. 不能出现幻觉
   2. 稳定
   3. 遵循指令
   4. 开源（方便私有化部署）
